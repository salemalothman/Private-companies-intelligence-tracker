<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server action | Untrusted email / password / confirm form input |
| recovery email link → /auth/confirm | token_hash + type arrive without a prior session |
| middleware auth gate | Decides reachability of unauthenticated + mid-recovery URLs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-pwd-01 | Information disclosure | requestPasswordReset | mitigate | Always return `{ sent: true }` and render a neutral message regardless of account existence or Supabase result — no account-existence oracle. |
| T-pwd-02 | Tampering (open redirect) | requestPasswordReset redirectTo | mitigate | redirectTo built server-side as `${siteUrl()}/auth/confirm`; never sourced from request/user input. |
| T-pwd-03 | Spoofing / replay | /auth/confirm verifyOtp | mitigate | Rely on Supabase single-use, time-limited recovery token via verifyOtp; on failure redirect to /login?error, never grant a session. |
| T-pwd-04 | Elevation of privilege | middleware /reset-password bypass | accept | The pre-approval-gate early-return is scoped to the exact path "/reset-password" and only lets a session-holder set a password; it grants no data access (all app data stays RLS + /pending gated). Rationale documented inline. |
| T-pwd-05 | Denial of service | resetPasswordForEmail spam | accept | Supabase applies built-in per-email rate limiting on its auth mailer; no app-level throttle added for this quick scope. |
| T-pwd-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed — uses existing @supabase/ssr, @supabase/supabase-js, lucide-react, and shipped ui/dialog. |
</threat_model>
